# VIviDaily PRD 讨论进度

> 最后更新：2026-04-11

---

## 一、产品定位

| 维度 | 内容 |
|------|------|
| **产品名称** | VIviDaily |
| **核心形态** | 日报形式的AI新闻聚合网页应用 |
| **触发方式** | Landing Page点击按钮触发爬取（爬取昨天一天热点） |
| **目标用户** | AI从业者、研究者、投资者、产品经理、技术观察者、前端/后端开发者、普通爱好者 |

### 当前实现基线说明

本阶段不再单独设计一套“正式产品页面”再落地，而是直接以现有前端原型仓库 `ViviDaily-github/` 作为实现基线继续开发。

- 文档中的页面结构、视觉方向、交互入口，以 `ViviDaily-github` 当前代码为准
- 后续功能实现优先在现有前端原型上补全，而不是重做一套新页面
- 如 PRD 与现有前端原型描述冲突，以“现有前端原型”版本优先，并同步更新文档

---

## 二、页面总览

| 页面 | 说明 |
|------|------|
| **Landing Page** | 入口页，采用视频背景 + 顶部导航 + 居中主视觉，点击 `Get Started` 进入抓取与加载流程 |
| **Feed Page** | 主页面，顶部导航 + 左侧分类栏 + 中间新闻流 + 右侧 Top5/AI Summary |
| **收藏夹页面** | 复用主框架，在主内容区展示三列收藏卡片 |
| **登录弹窗** | 当前前端为居中 Modal，不再按左侧抽屉形态实现 |

## 五、数据流程（已确认）

```
用户点击"获取今日热点"
         ↓
  ┌─────────────────┐
  │ 今天是否已爬取？ │
  └─────────────────┘
         ↓
    ┌────┴────┐
    ↓         ↓
  【否】     【是】
    ↓         ↓
 清除旧数据   直接跳转
 爬取昨天数据
    ↓         ↓
 跳转Feed    跳转Feed
```

### 数据生命周期

| 数据类型 | 生命周期 |
|---------|---------|
| **新闻流数据** | 每天爬取时清除旧数据，只保留当天爬取的 |
| **收藏夹数据** | 永久保留，用户手动取消收藏才删除 |

### 示例时间线

| 时间 | 用户行为 | 新闻流数据 | 收藏夹数据 |
|------|---------|-----------|-----------|
| 4月11日 09:00 | 点击"开始" | 爬取4月10日热点 | 空 |
| 4月11日 09:30 | 收藏2条新闻 | 4月10日热点 | 2条收藏 |
| 4月12日 10:00 | 点击"开始" | 清除旧数据，爬取4月11日热点 | 2条收藏（保留） |

## 六、功能清单（更新后）

### P0 - MVP核心

| 编号 | 功能 | 说明 |
|------|------|------|
| F1 | Landing Page入口 | 一个按钮触发爬取昨天一天热点，跳转到Feed Page |
| F2 | 数据采集 | 爬取预设信息源的昨天一天热点新闻 |
| F3 | AI智能处理 | 标题、摘要、标签提取、热度评分。标签体系分为来源标签、分类标签、主题标签三类；其中分类标签基于规则生成，主题标签采用 AI 辅助提取，优先从预定义标签集合中选择 1 到 3 个；如果预定义标签都不合适，允许 AI 额外提炼 1 个简短补充标签。 |
| F4 | 本地数据库 | 每天爬取时清除旧数据，只保留当天；收藏永久保留 |
| F5 | 新闻流列表 | Feed Page中间栏，卡片式展示 |
| F6 | 今日Top5 | 右侧栏上方，热度最高的5条。Top5 榜单基于当天通过过滤后的新闻候选池生成。系统先对候选新闻计算 `hotScore`，再按分数从高到低排序，去重后取前 5 条，作为今日热门榜单。 |
| F7 | 今日AI总结 | 右侧栏下方，AI生成的热点总结（3点） |
| F8 | 菜单筛选 | Hot/商业/模型/产品/政策，点击筛选新闻流；其中 Hot 表示所有爬取的新闻总入口，其他菜单项为细分类筛选。该分类同时作为分类标签写入新闻卡片。 |
| F9 | 收藏功能 | 卡片内⭐按钮，收藏/取消收藏 |
| F10 | 收藏夹页面 | 三列卡片展示已收藏内容 |
| F11 | 搜索功能 | 中间栏顶部搜索框，关键词搜索 |
| F12 | 查看原文 | 点击跳转原文链接 |
| F13 | 分享功能 | 分享新闻 |
| F14 | 导入Notion | 用户授权后写入自己的Notion |
| F15 | 登录/注册 | 居中弹窗打开，邮箱密码方式 |

### P1 - 重要功能

| 编号 | 功能 | 说明 |
|------|------|------|
| F16 | 用户状态 | 登录后左侧菜单栏显示用户头像 |
| F17 | 收藏同步 | 登录用户收藏数据可同步（待确认） |

## 十、Notion 字段对齐（2026-04-11）

当前 PRD 已按最新 Notion 数据库字段结构同步，后续采集、AI 处理、入库与前端展示都以这一版字段口径为准。

| 字段名 | 类型 | 用途 | 当前规则 |
|------|------|------|------|
| `Title` | Title / 文本 | 新闻标题 | 使用清洗后的标题写入，字段名末尾不再保留空格 |
| `Source` | Select / 文本 | 新闻来源 | 使用标准化来源名写入，字段名末尾不再保留空格 |
| `Category` | Select | 主分类 | 单选字段，对应页面分类体系 |
| `Topics` | Multi-select | 主题标签 | 多选字段，由 AI 从预定义主题集合中提取，必要时可补 1 个短标签 |
| `HotScore` | Number | 热度分数 | 数值字段，用于全量新闻排序与 Top5 榜单生成 |

### 已移除字段

- `Impact`
- `Audience`

### 字段生成方式

| 字段 | 生成方式 |
|------|------|
| `Title` | 直接取 RSS 标题并做清洗，去除多余空格与噪音格式 |
| `Source` | 根据采集源配置映射为统一来源名称 |
| `Category` | 基于规则和关键词命中结果，给每条新闻确定 1 个主分类 |
| `Topics` | 将标题和摘要输入 AI，优先从预定义主题标签集合中选择 1 到 3 个；若没有合适项，可补充 1 个简短主题标签 |
| `HotScore` | 基于重点关键词命中、来源权重、热点聚合信号、重复出现情况等规则综合计算 |

### 与页面功能的对应关系

- Feed 卡片展示字段：`Source`、`Topics`、`Category`、发布时间、摘要
- 左侧菜单筛选：主要基于 `Category`
- `Hot` 菜单：表示当天抓取的全部新闻总入口，不是单独主题
- Top5 榜单：从当天抓取的全部新闻中按 `HotScore` 排序、去重后取前 5 条
- Notion 导出：写入当前字段结构，不再写入 `Impact`、`Audience`

## 标签体系说明

当前新闻卡片与新闻流使用三类标签：

| 标签类型 | 作用 | 示例 |
|------|------|------|
| **来源标签** | 标识新闻来自哪个信息源 | 量子位、36氪、雷锋网 AI、InfoQ |
| **分类标签** | 用于左侧菜单筛选与新闻流归类；其中 Hot 表示所有爬取新闻的总入口 | Hot、商业、模型、产品、政策 |
| **主题标签** | 用于概括新闻核心技术或事件主题，由 AI 从预定义标签集合中选择 1 到 3 个 | LLM、Agent、AIGC、多模态、开源、融资、推理 |

标签使用规则：

- 来源标签用于帮助用户快速识别文章来源
- 分类标签与左侧菜单筛选逻辑保持一致，其中 Hot 表示所有爬取新闻的总入口，不是单独的一类新闻主题
- 分类标签通过规则与关键词匹配生成，用于保证筛选逻辑稳定
- 主题标签采用 AI 辅助提取：系统将标题和摘要提供给 AI，优先要求其从预定义标签集合中选择 1 到 3 个

预定义主题标签集合：

- LLM
- Agent
- AIGC
- 多模态
- 开源
- 融资
- 推理
- AI产品
- 模型发布
- AI芯片
- 自动驾驶
- 机器人
- AI应用
- AI安全
- 政策监管
- 数据工程
- RAG
- 算力
- 云平台
- 编程助手

主题标签补充规则：

- 如果预定义标签集合中没有合适标签，允许 AI 额外生成 1 个简短主题标签作为补充
- 补充标签应控制在 2 到 6 个中文字符，或 1 到 3 个英文词
- 补充标签必须是高概括性的主题词，不能写成长句
- 补充标签不得与分类标签重复
- 补充标签不得使用过于宽泛的表述，例如“新闻”“热点”“AI”“技术”
